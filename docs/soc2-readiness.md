# Spec: SOC 2 Technical Readiness Evidence

## Objective

Extend ClusterProof so a platform or security team can show which versioned
technical checks ran, what inputs were assessed, which findings were observed,
and whether the resulting evidence bundle still has integrity.

ClusterProof does not perform or certify a SOC 2 examination. It assesses only
the Kubernetes and software-supply-chain mechanisms visible to the scanner.
Organizational controls such as personnel, vendor management, policy approval,
and incident exercises remain outside the scan scope.

The output vocabulary deliberately avoids `compliant` and `passed`:

- `attention_required`: one or more mapped findings were observed.
- `no_findings_observed`: mapped rules ran without producing a finding.
- `not_assessed`: no rule in the selected ruleset assessed the reference.

## Ruleset Contract

The Community binary ships a data-only catalog named `clusterproof-default`,
versioned independently from the CLI. Every native rule definition includes:

- Stable rule ID and title.
- Category and high-level control references.
- Relationship to a named external source: `aligned` or `supplemental`.
- Source name, version, and official HTTPS URL.

The built-in Kubernetes rules are PSS-oriented but are not a complete
implementation of every Pod Security Standard control. The catalog must describe
the exact alignment instead of claiming PSS conformance.

External policy code is never downloaded or executed by default. ClusterProof may
import a bounded local JSON `PolicyReport` generated by another policy engine.
Initial support is limited to `wgpolicyk8s.io/v1alpha2`; OpenReports remains alpha
and can be added behind a separate adapter when its contract stabilizes.

## Evidence Bundle Contract

`clusterproof scan ... --evidence-dir DIR` creates a new private directory with:

- `scan.json`: canonical findings and summary.
- `ruleset.json`: exact native catalog evaluated by the scan.
- `controls.json`: native assessed rules and observed finding-rule counts by
  reference. External findings never masquerade as native assessed rules.
- `metadata.json`: tool, target, timestamp, input inventory, and scope notice.
- `bundle-manifest.json`: exact file list, byte sizes, and SHA-256 hashes.

`clusterproof evidence verify DIR` validates the bundle without contacting a
network service. Verification rejects:

- Missing, extra, duplicate, non-regular, or symlinked files.
- Unsafe paths or unsupported hash algorithms.
- Oversized manifests, files, or aggregate bundle input.
- Invalid hash encodings, byte sizes, or content hashes.

The integrity manifest detects changes relative to itself. It is not a digital
signature and does not establish who created the bundle. Long-term custody should
store the manifest in an immutable or externally signed system.

## Public Commands

```text
clusterproof scan [PATH | --kubeconfig PATH] \
  --evidence-dir evidence \
  --policy-report-json policy-report.json

clusterproof ruleset show [--format table|json]
clusterproof evidence verify DIR
```

`--policy-report-json` is optional and can be used with repository or cluster
scans. It imports results only; it does not install or invoke Kyverno,
Gatekeeper, or another policy engine.

## Project Structure

```text
internal/rules/         Native evaluation plus immutable rule catalog
internal/policyreport/  Bounded external PolicyReport JSON normalization
internal/evidence/      Readiness coverage, bundle writer, and verifier
cmd/clusterproof/       Ruleset, evidence, and scan CLI contracts
```

## Testing Strategy

- Catalog tests prove every native finding ID is registered once and every source
  URL is official HTTPS metadata.
- Coverage tests prove mapped references appear even when no findings are
  observed and never use a compliance claim.
- PolicyReport tests cover pass/skip omission, fail/warn/error normalization,
  hostile text cleaning, malformed JSON, unsupported API versions, and size caps.
- Bundle tests cover successful verification, tampering, extra files, symlinks,
  duplicate entries, unsafe paths, malformed hashes, and resource limits.
- CLI tests cover ruleset JSON, evidence verification exit codes, and combined
  scan enrichment.

## Boundaries

- Always: keep scans read-only, record catalog versions, bound external input,
  use private file modes, and preserve deterministic ordering.
- Ask first: add a rule execution engine, download policy bundles, sign evidence,
  store evidence remotely, or ingest licensed control text.
- Never: claim certification, emit `compliant`, bundle AICPA criteria text, bundle
  commercial CIS content without permission, execute imported policy code, or
  silently accept an unverifiable evidence file.

## Success Criteria

1. A secure scan lists assessed references as `no_findings_observed`, not
   `compliant`.
2. A finding changes only its mapped references to `attention_required`.
3. Every native finding ID is covered by the versioned catalog.
4. A bounded `wgpolicyk8s.io/v1alpha2` JSON report can enrich the canonical report
   without executing external code.
5. `clusterproof evidence verify` succeeds on an untouched bundle and fails on
   any added, removed, symlinked, or modified file.
6. Reports and bundles contain no AICPA criteria descriptions or secret values.
7. Tests, vet, build, race checks, fixture scans, and release packaging pass.
